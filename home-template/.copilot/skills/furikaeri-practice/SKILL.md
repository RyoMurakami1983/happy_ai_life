# ふりかえりプラクティス

セッション終了前に学びを整理し、**T（つぎにやること）** を決めて次のセッションに繋ぐ構造化ワークフロー。Quick モード（YWT、3-5分）を既定とし、Deep モード（KPT＋深掘り）はオプションで選べる。

> **命名について**: 「ふりかえり」はひらがな表記が正式名称です。漢字の「振り返り」とは意図的に区別しています。「ふりかえり」は構造化された改善プラクティスを指し、単なる「過去を振り返る」行為とは異なります。

## こんなときに使う

- セッション終了前に学びを残したい（「ふりかえり」と入力）
- コーディングセッションや複数ターンの会話を終えるとき
- PR レビューのフィードバック対応が完了したとき
- 根本原因の分析や改善アクションの具体化が必要なとき（→ Deep モード）

## 関連スキル

- **`git-commit-practices`** — コミット履歴を学習資産に
- **`github-pr-workflow`** — PR作成とIssue close
- **`github-issue-intake`** — 改善項目をIssueとして起票
- **`knowledge-capture`** — 公開時の匿名化ゲート

---

## 依存関係

- 必須なし（会話のみのワークフロー）
- オプション: GitHub CLI (`gh`) — T の Issue 化に使用

## コア原則

1. **立ち止まる** — 次のタスクに急ぐ前に、一度止まって観察する (温故知新)
2. **小さくカイゼン** — 少しずつ変える。小さな一歩が複利で効く (継続は力)
3. **知識を外に出す** — 暗黙の学びをチームの資産にする (成長の複利)
4. **事実ベース** — 事実から出発する。思い込みや責任追及ではなく (ニュートラル)
5. **型が洞察を生む** — KPT・YWT・5つのなぜ・SMARTという型が深い気づきを引き出す (基礎と型)

## ふりかえりの3つの目的

1. **立ち止まる** — 変化のきっかけをつくる。周りをみる。
2. **成長を加速させる** — 高頻度のコミュニケーションで悩み・問題を自然に共有し、即解決する。
3. **プロセスをカイゼンする** — 「価値をどう生み出しているか」に着目する。変更は小さく少しずつ。

## セッション接続（Step 0）

どのモードでも最初に行う。前回の T を起点に、セッション間を繋ぐ。

**エージェントの動作**:
1. `session-context.instructions.md`（session-start hook が自動注入）から前回の T セクションを探す
2. T が見つかった場合、各項目の達成状況を確認する:
   - ✅ 完了 — 今回のセッションで達成した
   - ➡️ 継続 — まだ途中、今回も取り組む
   - ❌ 破棄 — 状況が変わり不要になった
3. T が見つからない場合（初回セッション等）はスキップして Step 1 へ

ユーザーへの問いかけ:
> 「前回のセッションで決めた T（つぎにやること）はこちらです。達成状況を教えてください。」

```markdown
## 前回の T 達成確認

| # | 前回の T | 状況 |
|---|---------|------|
| 1 | 実環境でセッション終了→再開のサイクルを検証する | ✅ 完了 |
| 2 | sessionStart でコンテキスト注入が動いているか確認する | ➡️ 継続 |
```

> **Values**: 温故知新 / 継続は力

## Quick モード（既定 3-5分）

「ふりかえり」で起動する既定のワークフロー。YWT で素早く整理し、T に集中する。

### Step 1: Y（やったこと）を整理する

セッションのコンテキスト（会話履歴、ツール使用、ファイル変更）から、何をしたかを自動整理する。

**エージェントの動作**:
1. セッション中の会話を振り返り、主要なアクティビティを箇条書きで列挙する
2. ファイルの作成・変更があればそれも含める
3. 整理した Y をユーザーに提示し、抜け漏れがないか確認する

```markdown
### Y（やったこと）
- ふりかえりスキルの統合計画を作成した
- session-end.js と session-start.js のデータフローを分析した
- Hook 互換のファイルフォーマットを設計した
```

ユーザーへの問いかけ:
> 「セッションの Y（やったこと）を整理しました。追加・修正はありますか？」

> **Values**: 温故知新 / ニュートラル

### Step 2: W（わかったこと）を聞く

W はセッション中の気づき・学び・発見。機械的に抽出できないため、ユーザーに聞く。

ユーザーへの問いかけ:
> 「このセッションで気づいたこと、学んだことはありますか？ 技術的な発見、設計判断、プロセスの改善点など、何でも構いません。」

**ヒントとして提示する観点**:
- 今まで知らなかった技術的な事実
- 「こうすればよかった」という判断
- 予想外に時間がかかった/かからなかったこと
- 再利用できそうなパターンやアプローチ

```markdown
### W（わかったこと）
- スキルと hook のファイル名が不整合だと、データフローが途切れる
- session-start.js の isTemplateOnly() は5セクション全部を前提とする
- Quick モードでも Hook 互換フォーマットを守る必要がある
```

> **Values**: 基礎と型 / 成長の複利

### Step 3: T（つぎにやること）を決める

T は次のアクション。具体的で実行可能な項目に絞る。**ワークフローの主役**。

ユーザーへの問いかけ:
> 「次にやることを決めましょう。具体的なアクションを1〜3個お願いします。」

**ルール**:
- 最大3個に絞る（多すぎると実行されない）
- 「気をつける」「意識する」は T ではない。具体的な行動にする
- 必要なら GitHub Issue 化を提案する（Step 4 で対応）
- Step 0 で「➡️ 継続」とした項目は、引き続き T に含めてよい

```markdown
### T（つぎにやること）
- 統合後の SKILL.md を実環境でテストする
- session-end.js → session-start.js の完全なループを検証する
```

> **Values**: 継続は力 / 成長の複利

### Step 4: 保存 + Issue 化

#### 4a: `-session.md` に保存する

`.github/sessions/` に **Hook 互換フォーマット** で保存する。

**ファイル名規則**: `{YYYY-MM-DD}-{session-id-short}-session.md`

- `session-id-short`: セッション state フォルダ名の先頭8文字（取得できない場合は `manual`）
- **既存の `-session.md` がある場合**: W/T セクションを更新する（ヘッダや Y は session-end hook が管理するため上書きしない）
- **ない場合**: 新規作成する

**出力フォーマット**（Hook 互換）:

````markdown
# Session: {YYYY-MM-DD}
**Date:** {YYYY-MM-DD}
**Started:** {HH:MM:SS}
**Last Updated:** {HH:MM:SS}
**Project:** {プロジェクト名}
**Branch:** {ブランチ名}
**Worktree:** {cwd}

---

<!-- SESSION:SUMMARY:START -->
## セッション要約（YWT）

### Y（やったこと）
- {箇条書き}
<!-- SESSION:SUMMARY:END -->

### W（わかったこと）
- {箇条書き}

### T（つぎにやること）
- {箇条書き}

### Notes for Next Session
- {次のセッションへの申し送り}

### Context to Load
```
{次のセッションで読み込むべきファイルパス}
```
````

**エージェントの動作**:
1. `.github/sessions/` ディレクトリが存在しなければ作成する
2. **既存の session.md がある場合**: W/T/Notes/Context セクションのみを更新する（Y とヘッダは session-end hook が管理するため触らない）
3. **ない場合**: 上記フォーマットで新規作成する（Y を含む全セクションを書く。Y は session-end hook が後から events.jsonl ベースの内容で上書きするため、skill 側の Y は暫定版でよい）
4. 保存先パスをユーザーに伝える
5. git add はしない（`.github/sessions/` は通常 gitignore 対象）

> **Hook 互換の理由**: この構造を守ることで、session-end hook がヘッダと Summary を補強し、session-start hook が次セッションにコンテキストを注入する完全なループが成立する。`isTemplateOnly()` は Y/W/T/Notes for Next Session/Context to Load の5セクションが全てプレースホルダーの場合のみテンプレート扱いするため、skill が内容を埋めれば自動的に次セッションに注入される。

#### 4b: T の Issue 化を提案する（オプション）

T の中に追跡が必要な項目があれば、GitHub Issue への変換を提案する。

ユーザーへの問いかけ:
> 「T の中で GitHub Issue にしたい項目はありますか？」

- 「なし」ならスキップして終了
- 「あり」なら Issue タイトルと本文の案を提示し、`gh issue create` で作成
- 発行した Issue 番号をふりかえりノート側の T に追記する

### 完了

> 「ふりかえり完了です。`/exit` でセッションを終了できます。」

## Deep モード（オプション 10-15分）

Quick モードでは足りない場合に使う。「フルふりかえり」「KPTで」「深掘りしたい」で起動。

### いつ Deep モードを使うか

| 状況 | 理由 |
|------|------|
| 問題が多かった / 同じ問題が再発している | KPT で Keep/Problem を明確に分離 |
| 根本原因が不明 | 5つのなぜで掘り下げ |
| アクション項目を具体化したい | SMART 目標に落とし込み |
| ふりかえりプロセス自体を改善したい | メタふりかえり |
| ノウハウをスキル化したい | スキル化候補チェック |

### Deep Step 1: セッションストーリーを整理する

何が起きたかを時系列で整理する。事実に集中し、感情や判断は次のステップで扱う。

```markdown
## セッションストーリー

1. Issue #42（検索フィルタ追加）に着手
2. featureブランチを作成、テストファーストで実装（TDD）
3. 結合テストでAPIタイムアウトが発生
4. モック化に切り替え、テスト通過
5. PRレビューで2件の修正指摘
6. 修正してマージ完了
```

ユーザーへの問いかけ:
- 「今日は何に取り組みましたか？」
- 「このセッションの流れを教えてください。」

> **Values**: 温故知新 / ニュートラル

### Deep Step 2: KPT で分析する

セッションストーリーを KPT に分類する。

| カテゴリ | 説明 | 例 |
|---------|------|-----|
| **Keep** | うまくいったこと — 続けたい | TDDでタイムアウトを早期発見できた |
| **Problem** | うまくいかなかったこと — 修正が必要 | APIドキュメントが古かった |
| **Try** | 次に試したいこと | テスト設定にタイムアウト設定を追加 |

Try には2つの目的があります：
- **Keep強化** — すでにうまくいっていることをさらに伸ばす（← K と表記）
- **Problemカイゼン** — うまくいかなかったことを修正・防止する（← P と表記）

```markdown
## KPT

### Keep
- TDDワークフローで問題を早期発見できた
- アトミックコミットのおかげでレビューがスムーズだった

### Problem
- APIドキュメントが古くなっていた
- タイムアウトのデバッグに30分かかった

### Try
- (← K) テストセットアップにタイムアウト設定を追加する
- (← P) APIドキュメント更新を完了定義（Definition of Done）に含める
```

ユーザーへの問いかけ:
- 「続けたいこと（Keep）は何ですか？」
- 「困ったこと（Problem）は何でしたか？」
- 「次に試したいこと（Try）は何ですか？」

> **Note**: KPT のリストは必ずマークダウンテキストで会話に出力し、スクロールして見返せる形で残すこと。`ask_user` 等の選択UIは、Step 3の優先度選択など判断が必要な場面のみ使用する。

> **Values**: 基礎と型 / 成長の複利

### Deep Step 3: 重要項目をピックアップする

すべてが同じ重要度ではない。影響の大きい項目に絞り込む。

Step 2 の結果をまとめて提示し、ユーザーに1〜3項目を選んでもらう:

```markdown
## 優先項目（ドット投票方式）

KPTの結果から、最も重要な項目はどれですか？

1. 🔴 APIドキュメントが古かった（Problem）
2. 🟡 タイムアウト設定を追加（Try）
3. 🟢 TDDワークフローが有効だった（Keep）

→ ユーザー選択: #1（ドキュメント問題）と #2（タイムアウト設定）
```

ルール:
- 最大3項目に絞る
- 少なくとも1つはアクション可能な項目（Problem または Try）を含める
- Keep 項目は認識するが、アクションプランは不要

> **Values**: ニュートラル / 継続は力

### Deep Step 4: 深掘りする（必要な場合のみ）

根本原因が不明な項目には **5つのなぜ** を適用する。その後、アクション項目を **SMART目標** に落とし込む。

**5つのなぜ（例）**

```markdown
## 5つのなぜ: APIドキュメントが古かった

1. なぜドキュメントが古かった？
   → API変更時に誰も更新しなかった。
2. なぜ誰も更新しなかった？
   → PRチェックリストに含まれていなかった。
3. なぜチェックリストに含まれていなかった？
   → API変更の完了定義を決めていなかった。
4. なぜ完了定義がなかった？
   → API変更プロセスを形式化していなかった。
5. なぜ形式化していなかった？
   → 少人数チームで、全員わかっていると思い込んでいた。

根本原因: API変更プロセスが形式化されていなかった。
```

**SMART目標**

| 要素 | 内容 |
|------|------|
| **S**pecific（具体的） | PRチェックリストに「APIドキュメント更新」を追加 |
| **M**easurable（測定可能） | API関連PRの100%にドキュメント更新を含める |
| **A**chievable（達成可能） | チェックリスト1項目の追加 |
| **R**elevant（関連性） | ドキュメント陳腐化を直接防止 |
| **T**ime-bound（期限付き） | 次のスプリントまでに追加 |

> **Values**: 温故知新 / 基礎と型

### Deep Step 5: メタふりかえり + スキル化候補チェック

#### メタふりかえり

ふりかえりプロセス自体を改善する。1つだけ質問する:

> 「次のふりかえりで変えたいことは何ですか？」

例:
- 「ストーリーに時間をかけすぎた。分析を増やしたい」
- 「次回はQuickモードで十分だった」
- 「メトリクス（所要時間、コミット数）を含める」

```markdown
## メタふりかえり

次回の改善: Quick モードで十分な内容だった。Deep は再発問題の時だけ使う。
```

#### スキル化候補チェック

> 「今回のセッションで言語化されたノウハウをスキル化できるものはあるか？」

スキル化候補の例：
- その場で生み出したチェックリスト
- 繰り返す問題クラスへの対処パターン
- 他のコンテキストにも応用できる意思決定フレームワーク

アクション分岐：
- **新スキルが必要** → `skill` を起動
- **既存スキルの更新** → 該当スキルに対して `skill` を起動
- **候補なし** → Step 6 に進む

> **Values**: 基礎と型 / 成長の複利

### Deep Step 6: 保存 + Issue 化

Quick モードの Step 4 と同じ手順で保存する。追加で：

- KPT 分析結果は Notes for Next Session に要約を含める
- 優先した Try / Problem 項目を GitHub Issue に変換する
- SMART 目標がある場合は Issue 本文に含める

```markdown
## Action → Issue 対応

| Action (SMART) | Issue |
|---|---|
| PRチェックリストに「APIドキュメント更新」を追加（次スプリントまで） | #128 |
| テストセットアップへタイムアウト設定を追加（今週中） | Draft |
```

> **アクション種別による分岐**
> - **コーディング・プロセスの改善** → GitHub Issue を作成（本ステップ）
> - **skill・型・ワークフローの改善** → `skill` を起動して該当スキルを更新

> **完了ゲート**: Deep モードは Issue 化まで完了して初めて「完了」とみなす。

> **Values**: 継続は力 / 成長の複利 / 基礎と型

### Deep Step 7: Anonymization Gate（公開時のみ）

ふりかえり内容が公開スキルや共有ドキュメントに流れる可能性がある場合は、`knowledge-capture` の **Anonymization Checklist**（AC-1〜AC-4）を適用してから出力する。プロジェクト固有名、データ形式、業務ドメイン語、具体的な数値は一般化する。

> **Values**: ニュートラルな視点

## ベストプラクティス

- Quick モードは3〜5分、Deep モードは10〜15分を目安にする
- 問題解決型のセッションには Deep（KPT）、日常のセッション終了には Quick（YWT）
- Y は事実ベースで書く。感情や評価は W に入れる
- T は「次のセッションの最初にやること」レベルの粒度にする
- T は最大3個。5つ放置するより1つ完了する方がよい
- アクション項目はすぐに書き留める — 記憶に頼らない
- 前回の T を確認してから新しいふりかえりを始める

## よくある失敗

1. **Y を省略して W/T だけ書く**
   対策: 事実の土台がない学びは再現性が低い。必ず Y から始める。

2. **アクション項目が多すぎる**
   対策: 最大3項目に絞る。5つ放置するより1つ完了する方がよい。

3. **Try が曖昧**
   対策: SMART 目標に変換する（Deep モード Step 4）。「もっと気をつける」はアクションではない。

4. **犯人探しの Problem**
   対策: プロセスとツールに注目する。「〇〇さんがミスした」ではなく「プロセスがXを許容した」。

5. **前回の T を見返さない**
   対策: Step 0 で必ず確認する。session-start hook が自動注入するので手間はない。

6. **分析で終わってしまう**
   対策: Deep モードでは Issue 化まで回してループを閉じる。

## アンチパターン

- ふりかえりをステータスレポートにする（改善のためのもので、報告のためではない）
- 問題が起きたときだけふりかえる（良いセッションにも学びがある）
- Deep モードを毎回使う（Quick で十分なことが多い。重いと続かない）
- セッション終了後に思い出して書く（鮮度が落ちる。終了前に書く）
- アクション項目を書いてフォローアップしない

## Hook 連携仕様

### データフロー

```
session-start.js → session-context.instructions.md（前回 session.md を注入）
      ↓
furikaeri-practice skill → Step 0 で前回 T を確認
      ↓
furikaeri-practice skill → Step 4 で -session.md に保存
      ↓
session-end.js → 同じ -session.md のヘッダ更新 + Summary 追記
      ↓
session-start.js（次回）→ session-context.instructions.md に注入
```

### ファイル名規則

`{YYYY-MM-DD}-{session-id-short}-session.md`

- サフィックスは必ず `-session.md`（session-start.js の `findRecentSessions()` がこのパターンのみ検索するため）
- `session-id-short` はセッション state フォルダ名の先頭8文字

### セクション構造の互換要件

session-end.js と session-start.js が前提とするセクション:

| セクション | 書き手 | 必須 |
|-----------|--------|------|
| ヘッダ（Date, Started, Last Updated, Project, Branch, Worktree） | hook / skill | ✅ |
| `---`（SESSION_SEPARATOR） | hook / skill | ✅ |
| `<!-- SESSION:SUMMARY:START/END -->` 内の Y | hook（自動抽出） | hook が追記 |
| `### W（わかったこと）` | skill | ✅ |
| `### T（つぎにやること）` | skill | ✅ |
| `### Notes for Next Session` | skill | ✅ |
| `### Context to Load` + code fence | skill | ✅ |

> `isTemplateOnly()` はこの5セクション（Y/W/T/Notes/Context）が全てプレースホルダーの場合のみ true を返す。skill が W/T を埋めれば false → 次セッションに注入される。

### モード選択ガイド

| 状況 | モード | 理由 |
|------|--------|------|
| 日常のセッション終了 | Quick | 3-5分で T に集中 |
| 問題が多いセッション | Deep (KPT) | Keep と Problem を明確に分離 |
| 学びが多いセッション | Quick (YWT) | 発見したことに焦点 |
| 根本原因が不明 | Deep + 5つのなぜ | システム的な問題を掘り下げ |
| 初めてのふりかえり | Quick | よりシンプルで直感的 |

## FAQ

**Q: Quick と Deep、どちらを使うべき？**
A: 迷ったら Quick。Quick で十分なことがほとんど。Deep は再発問題や複雑なセッションのときだけ。

**Q: Quick モードにはどのくらい時間をかける？**
A: 3〜5分。それ以上かかるなら Deep モードに切り替える。

**Q: 毎回ふりかえりする必要がある？**
A: 毎セッション実施が理想だが、強制はしない。少なくとも「ふりかえり」と入力するだけで Quick が走る。

**Q: 前回の T が session-context.instructions.md にない場合は？**
A: session-start hook が前回の session.md を注入するが、7日以上前のセッションは対象外。その場合は `.github/sessions/` を直接確認するか、Step 0 をスキップして構わない。

**Q: `-furikaeri.md` で保存していた旧ファイルはどうなる？**
A: session-start.js は `-session.md` のみ検索するため、旧ファイルは次セッションに注入されない。必要なら手動で内容を `-session.md` に移行する。
