# Standard Design Workflow

仕様書を技術設計書に落とし込み、構造・言語適合・セキュリティの3面で検証するワークフローです。spec-workshop の後段、planner の前段に位置します。設計と計画を分離する理由は、設計の再検証や部分的なやり直しが計画に波及しないようにするためです。

## こんなときに使う

このスキルは次のようなときに使います:
- 仕様が固まり、コンポーネント構造やデータフローを設計したいとき
- 実装計画（planner）に渡す前に、構造判断とセキュリティ設計を確認したいとき
- architect の設計判断を言語固有の型で検証したいとき
- 設計レビューを構造・セキュリティの両面で実施したいとき

## ワークフロー: 設計検証

### ステップ 1 — 仕様の受け取りと前提確認

spec-workshop の成果物（仕様書）を受け取り、設計に進める状態かを確認する。

確認すること:
- 仕様書が存在し、主要な要件が記述されているか
- 対象の技術スタック（言語・フレームワーク・既存基盤）が明確か
- 非機能要件（性能、安全性、可用性）の言及があるか
- 未確定事項が残っている場合は、設計に影響するかを判断する

仕様の不足が設計を阻む場合は、spec-workshop に戻ることを提案する。

### ステップ 2 — 構造判断（architect）

`architect` agent に設計判断を委譲する。

architect への入力:
- 仕様書の要件（機能・非機能）
- 対象の技術スタックと既存基盤の情報
- 統合ポイント（外部システム、既存コンポーネントとの接点）

architect から受け取るもの:
- コンポーネントの責務と境界
- データフローと状態管理
- 各判断の Why とトレードオフ
- ADR（重要な判断の場合）
- 余白メモ（今決めなくてよいこと）

### ステップ 3 — 言語/技術検証（shihan）

architect の構造判断を、対象言語の型と品質基準で検証する。

- 対象技術に応じて適切な師範を選択する:
  - C# / .NET → `dotnet-shihan`
  - Python → `python-shihan`
  - TypeScript / Node.js → `typescript-shihan`
  - Skill / Agent 設計 → `skill-shihan`
- 複数言語にまたがる場合は、主実装言語の師範を優先する
- 技術制約が特に強い場合のみ、このステップを architect の前に先行させてよい

師範に確認すること:
- architect の構造が対象言語/フレームワークで自然に実装できるか
- 言語固有のパターンやイディオムとの不整合がないか
- 型システムや依存管理の観点から無理がないか

師範からフィードバックがあれば、architect の設計を微修正する。

### ステップ 4 — セキュリティ設計確認（security-review）

`security-review` agent に設計段階のセキュリティ確認を委譲する。

security-review への入力:
- architect の設計判断（コンポーネント構成、データフロー）
- 認証/認可の設計方針
- 外部入力の経路
- 機密データの扱い

確認してもらうこと:
- trust boundary が明確に定義されているか
- 認証/認可の設計方針が決まっているか
- 機密データの保存・転送方針が決まっているか
- 外部入力の検証方針が決まっているか

セキュリティ上の指摘があれば、設計に反映する。

### ステップ 5 — planner handoff サマリ生成

設計判断と検証結果を、planner が実装計画を作れる形にまとめる。

handoff サマリに含めるもの:
- **設計判断の要約**: コンポーネント構成、責務境界、データフロー
- **確定事項**: architect と shihan の検証を通過した判断
- **制約**: 技術的制約、セキュリティ要件、非機能要件
- **未決定事項**: 設計段階で保留した判断とその理由
- **リスク**: 実装時に注意すべきリスクと対応方針
- **ADR**: 重要な設計判断の記録（ある場合）

## 出力の型

### 設計書

```markdown
# 技術設計書: [プロジェクト/機能名]

## 概要
[何を設計したか、なぜこの構造にしたか — 1-2 文]

## コンポーネント構成
[architect の設計判断から]

## データフロー
[入力から出力までの流れ]

## 技術検証結果
[shihan のフィードバックと反映内容]

## セキュリティ設計
[security-review の確認結果]

## 設計判断の記録
[ADR がある場合]

## 余白メモ
[今は決めなくてよいこと、将来の拡張で検討すべきこと]
```

### planner handoff サマリ

```markdown
# Planner Handoff: [プロジェクト/機能名]

## 設計判断の要約
[コンポーネント、責務、データフロー — 箇条書き]

## 確定事項
[実装で前提にしてよい判断]

## 制約
[技術的制約、セキュリティ要件]

## 未決定事項
[保留した判断とその理由]

## リスク
[実装時の注意点]
```

## 注意点

- **planner の責務に踏み込まない**: 実行計画、フェーズ分割、実装順序はこの skill の範囲外です。handoff サマリを渡して planner に委譲します。
- **code-quality-review はここでは呼ばない**: コード差分が存在しない段階なので、品質レビューは実装後に行います。
- **師範を全員呼ばない**: 対象技術に関連する師範だけを選択します。技術スタックが不明な場合は architect に確認を取ります。
- **仕様の穴を設計で埋めない**: 仕様レベルの未確定事項を見つけたら、spec-workshop に戻ることを提案します。
