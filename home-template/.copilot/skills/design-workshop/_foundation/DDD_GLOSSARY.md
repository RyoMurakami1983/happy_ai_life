# DDD & Balanced Coupling 用語集

design-workshop の sub_skill で使う共通用語です。原語を添えつつ、日本語で統一した呼び方を定めます。

## DDD 戦略パターンの用語

| 原語 | 日本語マッピング | 定義 |
|---|---|---|
| Bounded Context | 区切られた文脈 | 特定のモデルが一貫して通用する範囲。文脈の外では同じ言葉が別の意味を持ちうる |
| Ubiquitous Language | 同じ言葉 | チームと利害関係者がコード・会話・文書で共有する統一された語彙 |
| Domain | 事業活動 / 事業領域 | ソフトウェアが解決しようとしているビジネスの活動領域 |
| Subdomain | 業務領域 | 事業活動をさらに分割した個々の業務の塊 |
| Domain Logic | 業務ロジック | 業務のルールや判断をコードで表現したもの |
| Domain Event | 業務イベント | 業務上の意味を持つ出来事。「注文が確定した」「在庫が減った」等 |
| Context Map | 文脈の地図 | 複数の区切られた文脈がどう関係し合うかを俯瞰する図 |
| Shared Kernel | モデルの共有 | 2つの文脈が合意のうえで共有するモデルの一部 |
| Anti-Corruption Layer | モデル変換装置 | 外部モデルを自分の文脈のモデルに変換する層。内部モデルの汚染を防ぐ |
| Separate Ways | 互いに独立 | 統合のコストが高すぎる場合に、文脈同士を切り離す判断 |

## DDD 戦術パターンの用語

| 原語 | 日本語マッピング | 定義 |
|---|---|---|
| Aggregate | 集約 | トランザクション境界を持つオブジェクトのまとまり。外部からはルートを通じてのみアクセスする |
| Value Object | 値オブジェクト | 同一性を持たず、値そのものが意味を持つオブジェクト（例: 金額、住所） |
| Entity | エンティティ | 同一性（ID）で区別されるオブジェクト。ライフサイクルを通じて追跡される |
| Repository | リポジトリ | 集約の永続化と検索を担う抽象。データアクセスの詳細を隠す |

## サブドメイン分類（変動性の評価基準）

| 分類 | 日本語 | 特徴 | 変動性 |
|---|---|---|---|
| Core Subdomain | 中核業務領域 | 競争優位の源泉。「面白い問題」を扱う | **高い** — 継続的に改善・最適化される |
| Supporting Subdomain | 支援業務領域 | 競争優位ではないが既製品もない。「退屈な問題」| **低い** — 安定すれば変更頻度は下がる |
| Generic Subdomain | 汎用業務領域 | 既に解決された問題（認証、決済等）| **機能は低い / 実装は可変** — プロバイダ変更の可能性を評価する |

## Balanced Coupling モデル固有の用語

| 用語 | 定義 |
|---|---|
| 統合強度（Integration Strength） | コンポーネント間で共有する知識の量と暗黙性。Intrusive > Functional > Model > Contract の4段階 |
| 距離（Distance） | 変更を同期するコスト。コード構造・組織構造・ランタイム依存の3面で評価する |
| 変動性（Volatility） | 変更が発生する確率。サブドメイン分類でビジネス観点から見積もる |
| バランスルール | `BALANCE = (統合強度 XOR 距離) OR NOT 変動性`。不均衡が問題になるのは変動性が高い場合 |
| 高凝集 | 高強度 + 低距離。共に変更される部品が近くにある状態（モジュラー） |
| 疎結合 | 低強度 + 高距離。離れた部品が少ない知識で連携する状態（モジュラー） |
| 密結合 | 高強度 + 高距離。離れた部品が多くの知識を共有する状態（分散モノリスのリスク） |
| 低凝集 | 低強度 + 低距離。無関係な部品が近くに同居する状態（泥団子のリスク） |

## 使い方

- sub_skill のワークフロー内で DDD 用語を使う際は、この用語集の日本語マッピングを優先する
- 利用者が DDD に不慣れな場合は、日本語マッピングと1行定義を添えて説明する
- 用語集は設計の共通語彙であり、特定のフレームワークの採用を強制するものではない
