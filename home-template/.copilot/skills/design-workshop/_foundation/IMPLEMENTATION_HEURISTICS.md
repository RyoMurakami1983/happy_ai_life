# 実装の経験則

サブドメイン分類（中核 / 支援 / 汎用）の結果から、業務ロジックの実装方法・技術方式・テスト方針を選ぶための判断ガイドです。Vlad Khononov の著作を参考にした独自のまとめです。

## 使い方

balanced-coupling-design ワークフローのステップ 1（サブドメイン分類）が完了した後に参照します。

1. 各モジュールのサブドメイン分類を確認する
2. 判断ツリーに沿って、業務ロジック実装方法・技術方式・テスト方針を選ぶ
3. 選んだパターンをステップ 2（モジュラーアーキテクチャ設計）とステップ 3（モジュール設計書）の入力にする
4. テスト方針をステップ 4（テスト仕様）の入力にする

## 判断ツリー

### 中核業務領域（Core）の場合

```
中核業務領域
│
├─ 金銭処理・分析・監査記録を扱う？
│   │
│   ├─ はい
│   │   業務ロジック: イベント履歴式ドメインモデル
│   │   技術方式:     CQRS
│   │   テスト方針:   ピラミッド型
│   │
│   └─ いいえ
│       業務ロジック: ドメインモデル
│       テスト方針:   ピラミッド型
│       │
│       └─ 永続化モデルは複数か？
│           ├─ はい → 技術方式: CQRS
│           └─ いいえ → 技術方式: ポートとアダプター
```

### 支援または汎用業務領域（Supporting / Generic）の場合

```
支援 / 汎用業務領域
│
├─ データ構造は複雑か？
│   │
│   ├─ はい
│   │   業務ロジック: アクティブレコード
│   │   テスト方針:   ダイヤモンド型
│   │   │
│   │   └─ 永続化モデルは複数か？
│   │       ├─ はい → 技術方式: CQRS
│   │       └─ いいえ → 技術方式: レイヤードアーキテクチャ（4層）
│   │
│   └─ いいえ
│       業務ロジック: トランザクションスクリプト
│       テスト方針:   逆ピラミッド型
│       │
│       └─ 永続化モデルは複数か？
│           ├─ はい → 技術方式: CQRS
│           └─ いいえ → 技術方式: レイヤードアーキテクチャ（3層）
```

### 組み合わせ早見表

| サブドメイン | 条件 | 業務ロジック | 技術方式 | テスト方針 |
|---|---|---|---|---|
| 中核 | 金銭処理・分析・監査記録あり | イベント履歴式ドメインモデル | CQRS | ピラミッド型 |
| 中核 | 上記以外 + 永続化モデル複数 | ドメインモデル | CQRS | ピラミッド型 |
| 中核 | 上記以外 + 永続化モデル単一 | ドメインモデル | ポートとアダプター | ピラミッド型 |
| 支援/汎用 | データ構造複雑 + 永続化モデル複数 | アクティブレコード | CQRS | ダイヤモンド型 |
| 支援/汎用 | データ構造複雑 + 永続化モデル単一 | アクティブレコード | レイヤード（4層） | ダイヤモンド型 |
| 支援/汎用 | データ構造単純 + 永続化モデル複数 | トランザクションスクリプト | CQRS | 逆ピラミッド型 |
| 支援/汎用 | データ構造単純 + 永続化モデル単一 | トランザクションスクリプト | レイヤード（3層） | 逆ピラミッド型 |

## テスト型の定義

| テスト型 | 焦点 | 構成 |
|---|---|---|
| ピラミッド型 | 単体テスト重視 | 単体テスト（多）→ 結合テスト（中）→ E2Eテスト（少） |
| ダイヤモンド型 | 結合テスト重視 | 単体テスト（少）→ 結合テスト（多）→ E2Eテスト（少） |
| 逆ピラミッド型 | E2Eテスト重視 | 単体テスト（少）→ 結合テスト（少）→ E2Eテスト（多） |

## 用語の定義

判断ツリーで使う用語の短い説明です。DDD 用語は `DDD_GLOSSARY.md` を参照してください。

### 業務ロジック実装方法

| 用語 | 説明 |
|---|---|
| イベント履歴式ドメインモデル | 状態の変化をイベントの時系列として記録し、現在の状態をイベントの再生で導出する。監査証跡や時点復元が自然に実現できる |
| ドメインモデル | 業務ルールと振る舞いをオブジェクト（エンティティ、値オブジェクト）として表現し、業務の複雑さをモデルで吸収する |
| アクティブレコード | データベースのテーブル構造と対応するオブジェクトに、CRUD 操作と簡単な業務ロジックを持たせる |
| トランザクションスクリプト | 業務操作を手続き的なスクリプト（関数）として実装する。単純な処理に向く |

### 技術方式

| 用語 | 説明 |
|---|---|
| CQRS | 読み取り（Query）と書き込み（Command）のモデルを分離する。複数の永続化モデルや複雑な読み取り要件がある場合に有効 |
| ポートとアダプター | 業務ロジックを中心に置き、外部との接点（DB、API、UI）をアダプターとして差し替え可能にする。ヘキサゴナルアーキテクチャとも呼ばれる |
| レイヤードアーキテクチャ（4層） | プレゼンテーション → アプリケーション → ドメイン → インフラストラクチャの4層で構成する。シンプルな関心の分離 |
| レイヤードアーキテクチャ（3層） | プレゼンテーション → アプリケーション → インフラストラクチャの3層で構成する。ドメイン層を独立させず、トランザクションスクリプトのような単純な業務処理を、アプリケーション層に寄せて扱うときに使う |

## 注意

- **経験則であり、機械的に適用するものではない**: 判断ツリーは出発点。実際のプロジェクトでは、チームのスキル、既存基盤、運用制約など文脈に応じて調整する
- **分岐条件は絶対的ではない**: 「データ構造は複雑か？」「金銭処理を扱うか？」の判断は文脈依存。迷ったら一段上（より堅牢な方）を選ぶのが安全
- **CQRS は段階的に導入できる**: 最初から完全な CQRS にする必要はない。読み取りモデルの追加から始めてもよい
- **テスト型は混合してよい**: モジュール内でもレイヤーによってテスト比率を変えるのは自然なこと
