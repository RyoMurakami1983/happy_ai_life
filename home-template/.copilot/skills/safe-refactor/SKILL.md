# safe-refactor

この skill は、安全な小規模リファクタリングのための手順です。

## 目的
- 振る舞いを変えずに可読性・保守性を改善する
- 差分を小さくし、レビューしやすくする
- 既存設計を尊重しつつ、重複や責務の混乱を減らす

## 手順
1. 変更対象の責務を 1 文で説明する。
2. 既存テストがあれば先に確認し、なければ最低限の振る舞い確認手段を用意する。
3. 変更理由を分類する。
   - 重複削減
   - 命名改善
   - 関数分割
   - 条件分岐整理
   - 依存の明確化
4. 1 回の変更で 1 つの関心事だけを扱う。
5. 公開 API や外部契約を変える場合は、互換性影響を明示する。
6. リファクタ後に、変更前後で何が同じで何が改善されたかを説明する。

## 優先する改善
- 長すぎる関数の分割
- 意味の薄い名前の改善
- 重複ロジックの集約
- エラー処理の明確化
- I/O と純粋ロジックの分離
- ドメイン概念の型 / 構造への引き上げ

## 避けること
- 無関係な整形や命名変更を大量に混ぜる
- 抽象化のための抽象化
- テストなしで大規模に動かす
- 既存の style や framework を無視する

## 出力形式
1. 現状の問題
2. 今回のリファクタ方針
3. 小さな変更ステップ
4. 互換性影響
5. 必要なテスト
6. 残課題
