# Resume

中断した仕様駆動開発を途中から再開する sub-skill です。成果物の存在状態から最も進んだフェーズを判定し、その次のフェーズから再開します。
ゴール駆動で使うため、最初に達成したいゴール、成功条件、確認手段を短く固定します。


## こんなときに使う

このスキルは次のようなときに使います:
- 中断した開発セッションを引き継ぎたいとき
- どこまで進んだか分からないとき
- 別のセッションで始めた作業を続けたいとき

## ワークフロー: 中断地点を判定して再開

### ステップ 1 — 成果物の存在を確認する

以下の順で成果物の存在を確認し、最も進んだ地点を判定します:

| 成果物 | 存在する場合の意味 | 次のフェーズ |
| --- | --- | --- |
| 実装中のコード + テスト + plan.md | 実装フェーズまで進んでいる（計画あり） | `impl-and-ship` へ（bootstrap 確認 → 残 slice の実装継続） |
| 実装計画（plan.md / todos） | 計画フェーズまで進んでいる（設計あり） | `impl-and-ship` へ（plan handoff 完了済みとして渡す） |
| 技術設計書 | 設計フェーズまで進んでいる（仕様あり） | `from-design` へ |
| 仕様書 | 仕様フェーズまで進んでいる | `from-spec` へ |
| 何もない | まだ始まっていない | `from-scratch` へ |

### ステップ 2 — 成果物の品質を確認する

存在する成果物が **完全な状態** か、それとも **不完全で戻す必要があるか** を判定します。

**plan.md がある場合**:
- 以下の必須セクションが揃っているか確認
  - Project Context（multirepository 構成を含む）
  - Phase Breakdown
  - Acceptance Criteria & Test Strategy
  - Risks & Mitigations
- 不足がある場合は、from-design に戻って PLAN mode を再実行することを提案

**設計書がある場合**:
- 以下が記述されているか確認
  - コンポーネント構成と責務境界
  - データフロー
  - セキュリティ設計の確認結果
- 不足がある場合は、design-workshop に戻ることを提案

**仕様書がある場合**:
- 以下が明記されているか確認
  - 機能要件と受け入れ条件
  - 対象の技術スタック
  - 非機能要件の言及
- 不足がある場合は、spec-workshop に戻ることを提案

### ステップ 3 — multirepository 環境での判定

複数リポが関連する場合、以下を確認してください:

- **plan.md の Project Context** に複数リポの構成と依存順序が記述されているか
- **各リポの plan.md** が存在するか（複数 plan 方式の場合）
- リポ間で進捗がズレていないか（例：backend は impl フェーズ、frontend は設計中）

リポ間でズレがある場合は、**進んでいないリポに合わせた段階から再開する** ことを提案します。

### ステップ 4 — 該当フェーズへ振り分ける

判定結果に基づいて、該当する skill / sub-skill に委譲します。

- **実装中のコード + テスト + 完全な plan.md** がある場合は `impl-and-ship` に委譲します。
- **完全な plan.md のみ** がある場合は `impl-and-ship` に委譲します（plan handoff 完了済みとして）。
- **完全な設計書のみ** がある場合は `from-design` に委譲し、PLAN mode → impl-and-ship へ進めます。
- **成果物が不完全** な場合は、1つ前のフェーズに戻ることを提案します（例：「plan.md が不完全 → from-design に戻す」）。

## 注意点

- **品質チェックは必須**: 存在確認だけでなく、必須セクションが揃っているか確認してください。不完全なまま進むと、実装フェーズで手戻りが大きくなります。
- **判定に迷ったらユーザーに聞く**: 自動判定が曖昧な場合は、ユーザーに現在地点と成果物の状態を確認します。
- **multirepository 環境での進捗ズレに注意**: 複数リポが関連する場合、全リポが揃う段階から再開することで、不整合を防ぎます。
