# セッション終了クイック YWT

セッション終了前に、**Y（やったこと）・W（わかったこと）・T（つぎにやること）** を素早く整理して `.github/sessions/` に保存する軽量ふりかえりスキル。

> **フルふりかえりとの違い**: 包括的な KPT/YWT ふりかえり（7ステップ、5つのなぜ、SMART、Issue化、Notion保存）が必要な場合は `furikaeri-practice` を使う。このスキルはセッション終了時の **クイック記録** に特化している。

## こんなときに使う

- コーディングセッションを終える前に、学びを残したい
- ユーザーが「ふりかえり」と入力した
- 短時間（3〜5分）で YWT を記録したい

## 依存関係

- なし（会話のみのワークフロー）
- オプション: GitHub CLI (`gh`) — T の Issue 化に使用

---

## ワークフロー

### Step 1: Y（やったこと）を整理する

セッションのコンテキスト（会話履歴、ツール使用、ファイル変更）から、何をしたかを自動整理する。

**エージェントの動作**:
1. セッション中の会話を振り返り、主要なアクティビティを箇条書きで列挙する
2. ファイルの作成・変更があればそれも含める
3. 整理した Y をユーザーに提示し、抜け漏れがないか確認する

```markdown
### Y（やったこと）
- session-continuity hook の MVP を実装した
- session-end.js で events.jsonl パーサーを作成した
- home-template と repo-template にテンプレートを配備した
```

ユーザーへの問いかけ:
> 「セッションの Y（やったこと）を整理しました。追加・修正はありますか？」

### Step 2: W（わかったこと）を聞く

W はセッション中の気づき・学び・発見。機械的に抽出できないため、ユーザーに聞く。

ユーザーへの問いかけ:
> 「このセッションで気づいたこと、学んだことはありますか？ 技術的な発見、設計判断、プロセスの改善点など、何でも構いません。」

**ヒントとして提示する観点**:
- 今まで知らなかった技術的な事実
- 「こうすればよかった」という判断
- 予想外に時間がかかった/かからなかったこと
- 再利用できそうなパターンやアプローチ

```markdown
### W（わかったこと）
- sessionStart の stdout 出力は公式で Ignored と明記されていた
- instructions.md 方式ならコンテキスト注入が確実に動く
- events.jsonl のパスマッチングは cwd ベースが最も安定
```

### Step 3: T（つぎにやること）を決める

T は次のアクション。具体的で実行可能な項目に絞る。

ユーザーへの問いかけ:
> 「次にやることを決めましょう。具体的なアクションを1〜3個お願いします。」

**ルール**:
- 最大3個に絞る（多すぎると実行されない）
- 「気をつける」「意識する」は T ではない。具体的な行動にする
- 必要なら GitHub Issue 化を提案する

```markdown
### T（つぎにやること）
- 実環境でセッション終了→再開のサイクルを検証する
- sessionStart でコンテキスト注入が動いているか確認する
```

### Step 4: ファイルに保存する

YWT を `.github/sessions/` に保存する。

**ファイル名規則**: `{YYYY-MM-DD}-{session-id-short}-furikaeri.md`

- `session-id-short`: セッション state フォルダ名の先頭8文字（取得できない場合は `manual`）
- 既存のセッションファイルがある場合は、そのファイル名からIDを流用する

**出力フォーマット**:

```markdown
# ふりかえり: {YYYY-MM-DD}

**日付**: {YYYY-MM-DD}
**プロジェクト**: {プロジェクト名}
**ブランチ**: {ブランチ名}

---

## Y（やったこと）
- {箇条書き}

## W（わかったこと）
- {箇条書き}

## T（つぎにやること）
- {箇条書き}

---

> furikaeri-ywt skill により自動生成
```

**エージェントの動作**:
1. `.github/sessions/` ディレクトリが存在しなければ作成する
2. 上記フォーマットで Markdown ファイルを書き出す
3. 保存先パスをユーザーに伝える
4. git add はしない（`.github/sessions/` は通常 gitignore 対象）

### Step 5: T の Issue 化を提案する（オプション）

T の中に追跡が必要な項目があれば、GitHub Issue への変換を提案する。

ユーザーへの問いかけ:
> 「T の中で GitHub Issue にしたい項目はありますか？」

- 「なし」ならスキップして終了
- 「あり」なら Issue タイトルと本文の案を提示し、`gh issue create` で作成

### 完了

> 「ふりかえり完了です。`/exit` でセッションを終了できます。」

## ベストプラクティス

- 3〜5分で終わらせる。深掘りが必要なら `furikaeri-practice` を使う
- Y は事実ベースで書く。感情や評価は W に入れる
- T は「次のセッションの最初にやること」レベルの粒度にする
- 毎セッション実施が理想だが、強制はしない

## アンチパターン

- Y を省略して W/T だけ書く（事実の土台がない学びは再現性が低い）
- T を5個以上書く（実行されない。3個以内に絞る）
- セッション終了後に思い出して書く（鮮度が落ちる。終了前に書く）
