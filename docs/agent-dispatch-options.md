# 4師範エージェントの進め方 विकल्प

この文書は、`dotnet-shihan` / `python-shihan` / `skill-shihan` / `typescript-shihan` をどう扱うかを比較するためのメモです。

## 背景

4師範エージェントは、`docs/PHILOSOPHY.md` の価値観を強く反映した「教える・広める・仕組み化する」実働部隊として追加された。
一方で、`planner` / `architect` / `tdd-guide` / `refactor` / `deep-review` は、単一責務のプロセスエージェントとして別の型を持っている。

ここで決めたいのは、4師範をどこまで実働化し、既存エージェントとどう分担するかである。

## 案A: 師範を助言専用レイヤに留める

### 中身

- 師範はレビュー、教育、言語固有の判断の説明に寄せる
- 実際の修正や進行は `deep-review` / `tdd-guide` / `refactor` / `build-resolver` に委譲する
- 師範は「何が良い型か」を教える役に徹する

### 良い点

- 最小権限に近い
- 既存構造を壊しにくい
- `architect` / `planner` / `tdd-guide` / `refactor` との衝突が少ない

### 気になる点

- 師範の「実働感」は弱い
- ユーザーから見ると、既存の専門エージェントとの差が薄くなる

## 案B: 師範をドメイン責任者、`tdd-guide` / `refactor` をプロセス責任者として分離する

### 中身

- 師範は「その言語や領域で何が自然か」を教える
- `tdd-guide` は Red-Green-Refactor の進行を担う
- `refactor` は振る舞い不変の整理を担う
- `.instructions.md` は静的ルールの正本、`*.agent.md` は動的判断の正本、`skills/` は手順の正本として分ける

### 良い点

- 実働化と境界整理の両立ができる
- `docs/PHILOSOPHY.md` の「教える」「仕組み化する」と相性がよい
- 将来、師範を増やしても分担しやすい

### 気になる点

- dispatch ルールの明文化が必要
- `deep-review` や `tdd-guide` との責務分界をはっきりさせる必要がある

### 推奨

**案B を推奨する。**

理由は、既存の process 系エージェントを壊さずに、4師範の強みだけを前面に出せるからである。

## 案C: 師範を各言語の総合エージェントにする

### 中身

- 師範が設計、実装、レビュー、改善まで広く担う
- ユーザーからは分かりやすい
- ただし `architect` / `tdd-guide` / `refactor` と衝突しやすい

### 良い点

- 入口が単純
- 個人利用では頼みやすい

### 気になる点

- God Agent 化しやすい
- 境界が溶けやすい
- 共有基盤としては保守しづらい

## まとめ

| 案 | ねらい | 向いている場面 |
|---|---|---|
| A | 安全に導入する | まずは壊さず試したい |
| B | 実働化と境界整理を両立する | 共有基盤として育てたい |
| C | 入口を単純にする | 個人利用で分かりやすさを優先する |

現時点の推奨は **案B** である。
