# Tetris-pilot完走で見えた母艦側の改善点

## Executive Summary

- 3-stack Tetris pilot を playable baseline まで完走したことで、前回より一段具体的に「母艦 repo 側で先に整えるべき contract」が見えた。
- 今回は repo 内に `SPEC.md` / `DESIGN.md` を増やさず、実装完走と比較結果を優先したため、改善点は GitHub Issue #47〜#52 に厳密な完了条件として集約した。
- 主な改善対象は `/sdd` の bootstrap checkpoint、`implementation-eval-gate` の runtime evidence、generator handoff、interactive app 向け harness contract である。

## Session Story

- `sdd` を入口に、target repo の bootstrap、3 トラックの scaffold、core gameplay、observability を順に揃えた。
- dotnet / python / typescript を最後まで実装し、deterministic seed、共通 state dump、共通 command runner の有効性と不足を比較した。
- 途中で Python の `uv run --project` より `--directory` が安定すること、未 `git init` repo は bootstrap 完了と言い切れないこと、runtime evidence の不足が language 差より支配的であることが見えた。
- その後、既存 Issue #47〜#50 の DoD を厳密化し、追加で Issue #51, #52 を起票した。

## Reflection

### Keep

- 3 トラックを最後まで実装したことで、文書上の推測ではなく実装ベースで改善点を特定できた。
- repo 側は `README.md` に比較 contract を寄せ、改善は GitHub Issue に分離したため、実装 repo と母艦 repo の責務を分けやすかった。
- 既存 Issue を重複起票せず、DoD の厳密化で再利用したのは履歴としても良かった。

### Problem

- 前回の Issue #47〜#50 は方向性として正しかったが、完了条件が「読める」「分かる」寄りで、実装後の検収観点としてはまだ弱かった。
- interactive app の比較では、algorithm より runtime loop / state observation / command contract の差が大きかったが、母艦側の標準手順にまだ十分反映されていなかった。
- `git init` 未実施や command 未固定のままでも bootstrap が済んだように見えてしまう余地があった。

### Try

- Issue #47〜#50 の DoD を「checkpoint が辿れる」「分岐が明文化されている」「必須項目がある」の形に更新する。
- Issue #51 で bootstrap 完了条件に `git init` と fixed command contract を追加する。
- Issue #52 で deterministic seed / state dump schema / command runner を interactive app harness の baseline contract として定義する。

### 5つのなぜ

1. なぜ今回さらに改善点が増えたか: 前回は eval/handoff の不足が見えた段階で止まり、今回は 3 トラックを playable baseline まで完走したため。
2. なぜ完走すると追加の改善点が見えたか: 実装後半で、比較の摩擦が gameplay そのものより bootstrap と観測 contract に集中していると確認できたため。
3. なぜ bootstrap と観測 contract が支配的だったか: language rules だけでは、Git 初期化状態、固定 command、deterministic state、runtime evidence を保証できないため。
4. なぜそれが母艦 repo 側の課題になるか: pilot repo ごとに毎回手で補うより、母艦の `/sdd`・checklist・harness contract に先回りして入れた方が再発を防げるため。
5. なぜ Issue 化で終わりではなく DoD を厳密化したか: 次回の改善セッションで「終わったかどうか」を迷わず判定できる状態にする必要があるため。

### SMART

- **Specific**: Issue #47〜#52 を入口に、bootstrap checkpoint、runtime evidence、handoff 必須項目、harness baseline contract を順に改善する。
- **Measurable**: 次回の pilot 開始時に、instructions 未配布・未 `git init`・command 未固定・runtime evidence 不足を前倒しで検出できる状態にする。
- **Achievable**: 変更対象は docs / checklist / skill / guidance の更新が中心で、Issue ごとに独立して進められる。
- **Relevant**: 今回の実装完走で見えた主要摩擦に直接対応している。
- **Time-bound**: 次の母艦改善セッションでは Issue #47 または #51 から着手する。
