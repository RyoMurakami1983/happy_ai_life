## モデルの概要（独自要約）

### 3 次元でカップリングを評価する

ソフトウェアコンポーネント間の結合を、以下の 3 次元で評価する:

1. **統合強度（Integration Strength）** — 共有する知識の量と暗黙性
   - Intrusive（最強）: 実装詳細への直接アクセス
   - Functional: 業務要件の共有・重複
   - Model: 業務モデルの共有
   - Contract（最弱）: 明示的な契約による統合
2. **距離（Distance）** — 変更を同期するコスト（技術的 + 組織的）
   - 同一メソッド → 同一クラス → 同一モジュール → 別サービス → 別システム
   - チーム構造も距離に影響する（Conway's Law）
3. **変動性（Volatility）** — 変更が発生する確率
   - DDD のサブドメイン分類で評価: Core=高、Supporting=低、Generic=機能低/実装可変
   - ビジネスドメインの観点から評価する（コミット頻度ではない）

### バランスルール

```
モジュラリティ = 統合強度 XOR 距離
バランス = (統合強度 XOR 距離) OR NOT 変動性
```

- **高強度 + 低距離** = 高凝集（モジュラー）
- **低強度 + 高距離** = 疎結合（モジュラー）
- **高強度 + 高距離** = 密結合（分散モノリスのリスク）
- **低強度 + 低距離** = 低凝集（泥団子のリスク）
- 変動性が低ければ、不均衡でも実害は小さい

### DDD パターンとの対応

| DDD パターン | カップリングの次元 |
|---|---|
| 境界付きコンテキスト | モデルカップリングの境界を定義 |
| コンテキスト統合パターン | 統合強度の管理 |
| 腐敗防止層 / 公開ホストサービス | Contract カップリングへの変換 |
| Aggregate / Value Object | 高強度 + 低距離 = 高凝集 |
| サブドメイン分類 | 変動性の評価基準 |
