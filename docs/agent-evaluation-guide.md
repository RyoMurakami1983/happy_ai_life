# Agent Evaluation Guide

この文書は、`home-template/.copilot/agents/` にある custom agent の評価と使い所を、
現在の repository 方針に沿って整理したものです。

前提は **Plan B** です。

- 師範系 agent は **ドメイン責任者**
- それ以外の specialist agent は **プロセス責任者** または **調査責任者**
- `skills/` は入口と手順、`*.instructions.md` は静的ルール、`*.agent.md` は動的判断を担う

関連文書:

- `docs/agent-dispatch-options.md`
- `.github/copilot-instructions.md`

## 先に結論

### いまの使い分け

| 分類 | agent | ひとことで言うと |
|---|---|---|
| 設計 | `architect` | 構造判断とトレードオフ整理の第一選択 |
| 計画 | `planner` | 複雑作業を安全な段取りに落とす第一選択 |
| 調査 | `deep-researcher` | 一次情報確認とベストプラクティス比較の第一選択 |
| レビュー（品質） | `code-quality-review` | 品質・回帰・非破壊性レビューの第一選択 |
| レビュー（セキュリティ） | `security-review` | セキュリティレビューの第一選択 |
| ビルド修正 | `build-resolver` | ビルドを最小差分でグリーンに戻す第一選択 |
| 性能 | `performance-optimizer` | 計測に基づく性能分析の第一選択 |
| 安全整理 | `refactor` | デッドコード削除・重複排除の第一選択 |
| TDD | `tdd-guide` | テストファーストで進める第一選択 |
| PyTorch | `pytorch-resolver` | PyTorch/CUDA/shape error の第一選択 |
| C#/.NET | `dotnet-shihan` | C#/.NET/WPF の型と品質基準の第一選択 |
| Python | `python-shihan` | Pythonic な型と品質基準の第一選択 |
| TypeScript | `typescript-shihan` | TypeScript/Node.js の型と品質基準の第一選択 |
| Skill | `skill-shihan` | skill の型と品質基準の第一選択 |

### 判断に迷ったときの目安

- 「どう設計するべきか」なら `architect`
- 「どういう順番で進めるべきか」なら `planner`
- 「まず何が事実か確認したい」なら `deep-researcher`
- 「この差分は品質的に大丈夫か」なら `code-quality-review`
- 「この差分はセキュリティ的に大丈夫か」なら `security-review`
- 「言語/skill として自然か」なら各 `*-shihan`
- 「テスト先行で実装したい」なら `tdd-guide`
- 「壊さずに削りたい/まとめたい」なら `refactor`
- 「ビルドが赤い」なら `build-resolver`
- 「遅い原因を測って特定したい」なら `performance-optimizer`
- 「PyTorch の実行時エラー」なら `pytorch-resolver`

## 評価の見方

この文書でいう「評価」は、優劣ではなく **責務との一致度** を表します。

- **非常に強い**: その依頼の第一選択
- **強い**: 明確に向いている
- **限定的**: 補助としては有効だが主担当ではない
- **向いていない**: 境界の外。別 agent に委譲したほうがよい

## Agent ごとの評価と使い所

### `architect`

- 評価: **非常に強い**
- 役割: 構造判断、トレードオフ整理、ADR 的な判断記録
- 向いている依頼:
  - 新機能の構造設計
  - コンポーネント境界の見直し
  - 大規模リファクタリング前の設計比較
  - 複数案の比較と採否判断
- 向いていない依頼:
  - 実装そのもの
  - 実行順序の詳細化
  - 小さなビルド修正
- 一言評価:
  - 「どう作るか」ではなく **「なぜその構造か」** を言語化する agent。設計の正本に近い。

### `planner`

- 評価: **非常に強い**
- 役割: 複雑作業を実行可能な段取りへ分解する
- 向いている依頼:
  - 実装計画の作成
  - 大きな変更のフェーズ分割
  - 影響範囲の洗い出し
  - 実装前の進め方整理
- 向いていない依頼:
  - 設計の是非判断
  - 直接のコード編集
  - ドメイン固有の品質判断
- 一言評価:
  - 余白を残した計画に強い。実装者を縛りすぎない計画がほしいときに効く。

### `deep-researcher`

- 評価: **非常に強い**
- 役割: 内部の source of truth と外部の公開知見を read-only で比較する
- 向いている依頼:
  - 実装前の事実確認
  - 現状のベストプラクティス調査
  - repository の過去判断と外部標準の照合
  - 曖昧な論点の前提整理
- 向いていない依頼:
  - 実装
  - 先回りした設計決め打ち
  - 調査結果をそのまま採用判断へ飛ばすこと
- 一言評価:
  - 「正しい前提づくり」に最も強い。設計や実装の前に噛ませる価値が高い。

### `code-quality-review`

- 評価: **非常に強い**
- 役割: ローカル変更を read-only で深くレビューし、品質・回帰・保守性の高信頼指摘に絞る
- 向いている依頼:
  - PR 前の品質レビュー
  - commit 前チェック
  - ロジック/非破壊性/回帰の確認
  - テスト不足の確認
  - source of truth 整合の確認
- 向いていない依頼:
  - セキュリティ脆弱性の詳細診断（→ `security-review`）
  - 実装作業
  - stylistic nit の大量列挙
- 一言評価:
  - 「本当に痛い品質指摘だけ返す」方向に寄っており、ノイズが少ない。

### `security-review`

- 評価: **非常に強い**
- 役割: ローカル変更を read-only でセキュリティ観点から深くレビューする
- 向いている依頼:
  - 認証・認可・外部入力を扱うコードの変更
  - シークレット混入の確認
  - 依存関係の脆弱性確認
  - trust boundary の検証
- 向いていない依頼:
  - コード品質やロジックの正確性（→ `code-quality-review`）
  - 実装作業
  - パターンカタログの網羅的確認
- 一言評価:
  - 攻撃シナリオに基づく具体的なリスク指摘に集中しており、誤検出が少ない。

### `build-resolver`

- 評価: **非常に強い**
- 役割: ビルドエラー、型エラー、依存解決エラーを最小差分で直す
- 向いている依頼:
  - コンパイルエラー修正
  - 型エラー修正
  - import/依存解決の破綻修正
  - CI の build failure の局所復旧
- 向いていない依頼:
  - ついでのリファクタリング
  - 新機能追加
  - 設計見直し
  - PyTorch 特有の runtime error
- 一言評価:
  - 「まず緑に戻す」ことに集中できるのが強み。無関係な改善を混ぜにくい。

### `performance-optimizer`

- 評価: **非常に強い**
- 役割: 計測に基づいてボトルネックを特定し、改善方針を示す
- 向いている依頼:
  - 遅い原因の切り分け
  - プロファイリング
  - I/O、CPU、メモリ、並行性の分析
  - リアルタイム制約やレイテンシ要件の検討
- 向いていない依頼:
  - 計測なしの当てずっぽうな最適化
  - 構造設計の決定
  - 直接の大改造
- 一言評価:
  - 「推測ではなく計測」を守らせる agent。性能課題を雑に扱わなくなる。

### `refactor`

- 評価: **非常に強い**
- 役割: デッドコード削除、重複排除、未使用依存整理を安全に進める
- 向いている依頼:
  - 未使用コードの削除
  - 重複実装の統合
  - 未使用依存の整理
  - 小さな段階での健全化
- 向いていない依頼:
  - 新機能開発
  - 設計刷新
  - テストが薄い領域での大規模削除
- 一言評価:
  - 「消してよいことを証明してから消す」型が明確で、安全な整理役として強い。

### `tdd-guide`

- 評価: **非常に強い**
- 役割: Red-Green-Refactor を厳守し、テストファーストで進める
- 向いている依頼:
  - 新機能の TDD 実装
  - バグ修正のテストファースト対応
  - カバレッジ向上
  - エッジケースや境界値の整理
- 向いていない依頼:
  - テストなしで先に実装したいケース
  - テストフレームワーク選定
  - アーキテクチャ設計の決定
- 一言評価:
  - 旧名から改名され、**ドメイン責任者ではなくプロセス責任者** であることが明確になった。

### `pytorch-resolver`

- 評価: **非常に強い**
- 役割: PyTorch 固有の runtime/CUDA/shape/gradient error を最小差分で直す
- 向いている依頼:
  - CUDA error
  - tensor shape mismatch
  - device mismatch
  - backward/gradient 計算エラー
  - DataLoader 周辺の実行時エラー
- 向いていない依頼:
  - モデル設計の刷新
  - ハイパーパラメータ最適化
  - 一般的な build/type error
  - TensorFlow/JAX 系の問題
- 一言評価:
  - 問題領域が狭いぶん精度を出しやすい。PyTorch 以外へは広げないのが正しい使い方。

### `dotnet-shihan`

- 評価: **強い**
- 役割: C#/.NET/WPF の型と品質基準を示すドメイン責任者
- 向いている依頼:
  - C# コードレビュー
  - .NET/WPF 実装の自然さの確認
  - モダン C# への改善
  - チーム標準の教育
- 向いていない依頼:
  - 構造設計の最終判断
  - TDD の進行管理
  - 安全な削除/統合の主担当
- 一言評価:
  - 「先生モード」と「求道者モード」があり、単なる lint ではなく **型を教えながら前進させる** 方向に強い。

### `python-shihan`

- 評価: **強い**
- 役割: Pythonic な設計と品質基準を示すドメイン責任者
- 向いている依頼:
  - Python コードレビュー
  - typing の方針整理
  - Pythonic でない実装の改善
  - エコシステム標準に沿った実務判断
- 向いていない依頼:
  - アーキテクチャ判断そのもの
  - TDD の進行責任
  - 汎用的なビルド復旧の主担当
- 一言評価:
  - Python の「自然さ」を保つ役として有効。PEP、typing、保守性の観点をまとめやすい。

### `typescript-shihan`

- 評価: **強い**
- 役割: TypeScript/Node.js の型と品質基準を示すドメイン責任者
- 向いている依頼:
  - TypeScript レビュー
  - strict mode 前提の型改善
  - `any` の削減
  - Node.js 実装のモダン化
- 向いていない依頼:
  - 構造設計の決定
  - TDD プロセスの主担当
  - 安全な削除作業の主担当
- 一言評価:
  - 型安全と実務バランスに強い。TypeScript の品質基準を会話可能な形で提示できる。

### `skill-shihan`

- 評価: **強い**
- 役割: skill の型と品質基準を示すドメイン責任者
- 向いている依頼:
  - skill のレビュー
  - skill の新規作成・改善方針の相談
  - validate/test を踏まえた品質基準の確認
  - skill 群のパターン整理
- 向いていない依頼:
  - skill workflow の入口そのもの
  - 計画立案や構造判断の代行
  - TDD/リファクタリング進行の主担当
- 一言評価:
  - この repository では特に重要。`skill` が手順を担い、`skill-shihan` が型と品質を担う分担がきれい。

## 師範系 agent の総評

4師範はどれも **「その領域で何が自然か」を示す役** として評価が高いです。
一方で、現在の方針では次を持ちません。

- 構造判断の最終責任
- 計画立案の責任
- Red-Green-Refactor の進行責任
- 安全な削除/統合の進行責任

つまり師範系 agent は、強い実働感を持ちながらも **万能 agent にはしない** のが現行方針です。

## 実務でのおすすめ呼び分け

### 例1: TypeScript の設計と実装を進めたい

1. `architect` で構造を決める
2. `planner` で段取りを切る
3. `tdd-guide` でテスト先行にする
4. `typescript-shihan` で型と品質を詰める
5. `code-quality-review` で差分確認する

### 例2: Python コードが遅い

1. `performance-optimizer` で計測と原因特定
2. 必要に応じて `python-shihan` で Pythonic な改善案を吟味
3. 変更後に `code-quality-review` で回帰確認

### 例3: skill を追加したい

1. `skill` で入口と手順を使う
2. `skill-shihan` で型と品質基準を詰める
3. 必要なら `planner` や `architect` を併用する
