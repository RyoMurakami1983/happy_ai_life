# 師範 Agent 運用モデル

**日付**: 2026-04-07
**ステータス**: 承認

---

## 背景

4 つの師範 agent（dotnet-shihan, python-shihan, typescript-shihan, skill-shihan）は、現在「先生モード」を既定とし、tools に `edit` を含まない。

仕様書は「求道者モードを既定にし、必要なら編集まで行う」ことを求めている。一方、Architect レビューは以下のリスクを指摘した:

- レビュー依頼でも改善実行に寄りやすくなる
- architect / tdd-guide / refactor との境界が溶ける
- 本文簡素化 (A)、既定モード変更 (B)、edit 権限追加 (C) を分離して評価すべき

## 判断

### 1. 3 要素の分離と段階適用

| 要素 | 判断 | 理由 |
|------|------|------|
| **(A) 本文簡素化** | 採用 | 統一構成で保守性向上。運用影響なし |
| **(B) 既定モード変更** | 条件付き採用 | 思考態度は求道者、行動権限は先生モードに分離 |
| **(C) edit 権限追加** | 条件付き採用 | 使用条件を厳格に定義した上で追加 |

### 2. 師範の責務境界

| agent | 責務 | 師範がやらないこと |
|-------|------|------------------|
| **師範（4 体）** | ドメイン固有の型・品質基準の提示、改善提案、コードレビュー | 構造判断の最終決定、計画立案、TDD の進行管理 |
| **architect** | 構造判断、トレードオフ分析、コンポーネント境界 | 言語固有のコーディング規約 |
| **planner** | 実装計画、段階分割、依存関係整理 | コードレビュー |
| **tdd-guide** | Red-Green-Refactor の進行管理 | 構造設計 |
| **refactor** | デッドコード削除、重複排除、安全な統合 | 新機能の設計判断 |

### 3. 2 つのモードの定義

#### 改善提案既定（既定モード）

仕様書の「求道者モード既定」を Architect 推奨に従い「改善提案既定」に緩和する。

- **思考態度**: 常にカイゼンの目線で見る（パターンの限界を見極め、より良い型を探る）
- **行動**: レビュー・提案・判断理由の提示に留める
- **edit の使用**: しない（このモードでは）
- **出力**: 改善案と理由を提示し、ユーザーの判断を待つ

#### 求道者モード（明示要求時）

- **トリガー**: ユーザーの明示依頼（「求道者モードで」「このパターンを改善して」等）、または他 agent からの handoff
- **行動**: 最小編集で改善を実践する
- **edit の使用**: 許可
- **制約**: repo-wide の build/test/validation を守る、変更理由と影響範囲を事前に提示する

#### 先生モード（明示要求時）

- **トリガー**: 「先生モードで」「レビューだけして」「教えて」等
- **行動**: レビュー専用または学習支援専用
- **edit の使用**: しない

### 4. 統一構成（8 セクション）

各師範 agent の本文は以下の構成に統一する:

1. **役割** — ドメインの専門性と責務の要約
2. **既定モード: 改善提案** — 思考態度と行動の定義
3. **求道者モード** — 明示要求時のふるまい
4. **先生モード** — 明示要求時のふるまい
5. **権限境界** — architect / planner / tdd-guide / refactor との分離
6. **改善時の優先順位** — (1) コンパイル不能 (2) 型・null 安全 (3) 例外・I/O (4) 責務過多 (5) readability (6) micro-optimization
7. **出力テンプレート** — 各モードの出力形式
8. **禁止事項** — 構造判断しない、仕様を勝手に拡張しない、build/test を無視しない、大規模再生成しない、Skill/MCP 接続詳細を本文に書かない

### 5. edit 権限の使用条件

| 条件 | edit 許可 |
|------|----------|
| ユーザーが「改善して」「直して」等を明示的に依頼 | ✅ |
| 他 agent（tdd-guide, refactor 等）からの handoff | ✅ |
| 通常のレビュー依頼 | ❌（提案に留める） |
| design-workshop 等から呼ばれた助言依頼 | ❌ |

### 6. 簡素化で削るもの

| 削除対象 | 理由 |
|---------|------|
| 「憲法」セクション（6 つの Values の列挙） | home instructions の正本に集約。agent は参照のみ |
| 「導入候補」「拡張候補」リスト | 師範の責務外。skill 管理やドキュメントに寄せる |
| 「守破離の長表」 | 概念は保持するが、詳細表は削除しコンパクトに |
| Skill の台帳的列挙 | 「関連 skill を参照する」と短く書く |
| MCP の説明 | 「決定論的なツール操作を優先する」とだけ書く |

### 7. 簡素化で残すもの

| 保持対象 | 理由 |
|---------|------|
| 品質基準（先生モードで使用） | agent の「型の提示」責務の核心 |
| レビューチェックリスト | 実用上の価値が高い |
| SLOP-001 等の固有品質観点 | ドメイン固有の知見 |
| WPF/MVVM の観点（dotnet-shihan） | 専門性の核心 |

## 根拠

- **Architect レビュー（2026-04-07）**: 「求道者を既定の思考態度にするのはよいが、既定の行動権限まで広げない」
- **仕様書の意図**: 師範を「単なる講師」ではなく「改善を実践する求道者」にしたい
- **両立方針**: 思考態度と行動権限を分離することで、仕様書の意図と Architect の慎重な推奨を両立させる

## トレードオフ

- **利点**: 師範がより積極的に改善の目線を持ちつつ、意図せぬコード変更のリスクを抑える
- **リスク**: 「改善提案既定」が中途半端に感じられる可能性（求道者モードの明示要求が面倒）
- **緩和策**: ユーザーが頻繁に求道者モードを使う場合、将来的に既定の行動権限を広げることを検討する
