# Custom Agent を作成する

この skill は、`docs/PHILOSOPHY.md` の思想を根幹に据えた custom agent（`.agent.md`）を
作成するためのワークフローです。参考となる既存実装をそのまま複製するのではなく、
自分たちの哲学に合わせて再設計する過程を案内します。

## こんなときに使う

- 新しい custom agent を `plugins/happy-ai-life/agents/` に追加したいとき
- 外部の agent 定義を参考にしつつ、自分の思想に合わせてカスタマイズしたいとき
- 既存の skill / instructions と整合する新エージェントを作りたいとき
- agent の役割定義や哲学的原則の選び方に迷っているとき

## ワークフロー: エージェントを作成する

### ステップ 1 — 役割を定義する

エージェントが「何を判断し、何を出力するか」を1文で定義する。
この1文が曖昧だと、エージェントの責務が他のエージェントや skill と重複しやすくなる。

確認すること:
- 既存エージェントとの責務の境界（重複しないか）
- このエージェントは read-only か、それとも実行権限が必要か
- 出力は判断・計画・レビュー・生成のどれに該当するか

### ステップ 2 — 哲学的原則を選ぶ

`docs/PHILOSOPHY.md` の Values から、このエージェントの判断に影響する原則を3〜5つ選ぶ。
選んだ原則はエージェント固有の言葉に落とし込み、そのまま引用しない。

PHILOSOPHY.md の Values:

| Value | 核心 | エージェント設計での活用例 |
|-------|------|------------------------|
| 温故知新 | 過去を学んでから提案する | 既存構造を調べるステップを設ける |
| 継続は力なり | コツコツ積み上げる | 段階的プロセスを設計する |
| 基礎と型 | 確立されたパターンを優先する | 既存パターンからの逸脱に Why を求める |
| 成長の複利 | 学びが残る構造にする | 出力に Why を必須化する |
| ニュートラル | 偏らず普遍的に | 技術中立を保つ、前提を明示する |
| 余白の設計 | 変化の余地を守る | 「今決めなくてよいこと」を分離する |

### ステップ 3 — モデルを選定する

エージェントの性質に合わせてモデルを選ぶ。最終判断はユーザーに委ねる。

推奨の考え方:

| エージェントの性質 | 推奨モデル | 理由 |
|-------------------|----------|------|
| 緻密な分析・長い推論が必要 | `claude-opus-4.6` | 深い推論と長文コンテキストに強い |
| 構造化・網羅的な検証が必要 | `gpt-5.4` | 緻密な構造分析と体系的推論に強い |
| 高速な応答・軽量タスク | `claude-sonnet-4.5` / `gpt-5.4-mini` | コスト効率と応答速度のバランス |

ユーザーに推奨を提示したうえで、選択を確認する。

### ステップ 4 — ツールを決める

エージェントの責務に応じて最小限のツールを選ぶ。

| 責務 | 推奨ツール | 理由 |
|------|----------|------|
| 分析・判断のみ（read-only） | `read`, `search` | 意図しない変更を防ぐ |
| 分析＋コマンド実行が必要 | `read`, `search`, `execute` | git diff 等のコマンドが必要な場合 |
| 分析＋最小編集が必要 | `read`, `search`, `edit` | 小さな修正や雛形更新までを agent に委ねる場合 |
| 分析＋コマンド実行＋最小編集が必要 | `read`, `search`, `execute`, `edit` | 検証しながら最小編集まで完結させる場合 |

Copilot CLI の custom agent では `edit` も公式に利用できますが、常時付けるのではなく責務に応じて最小権限で選びます。迷ったら少ない方を選び、必要になってから追加する。

### ステップ 5 — agent.md を作成する

`references/agent-template.md` を基に、ステップ1〜4の結果を埋め込んで作成する。

配置先: `plugins/happy-ai-life/agents/<name>.agent.md`

作成時のチェック:
- frontmatter が既存エージェントと同じフィールド構成か
- `description` に `Use when:` のトリガー句があるか
- 「役割」「非責務」「原則」「プロセス」「出力の型」「完了条件」が揃っているか
- 原則の説明が「なぜ効くか」を prose で書いているか（MUST/NEVER 列挙でないか）

### ステップ 6 — 既存エージェント群との整合性を確認する

既存の agent corpus がある場合は `plugins/happy-ai-life/agents/` 配下のエージェントと比較し、
frontmatter 形式、責務境界、文体の一貫性を確認する。Phase 2 以降のように corpus が空なら、
template と instructions との整合を優先する。

## 早見表

| 段階 | 何を決めるか |
|------|------------|
| 役割定義 | 1文の責務、既存との境界、read-only か否か |
| 原則選択 | PHILOSOPHY.md から 3〜5 原則、エージェント固有の言葉に変換 |
| モデル選定 | 推奨を提示、ユーザーが最終判断 |
| ツール選定 | 最小権限で選ぶ |
| 作成 | template 適用、frontmatter・構成チェック |
| 整合性確認 | 既存エージェント群との一貫性 |

## 注意点

- **参考をそのままコピーしない**: 外部の agent 定義を参考にする場合は、構造と着想だけを取り込み、
  哲学・文体・原則は PHILOSOPHY.md から再設計する。同じにしないことが前提。
- **パターンカタログにしない**: エージェントの本文にパターンや技術を網羅的に列挙しない。
  必要なパターンは実行時に判断する。
- **原則を引用で済ませない**: PHILOSOPHY.md の文言をそのまま貼るのではなく、
  エージェントの領域に合わせて「なぜこの原則がここで効くか」を説明する。
- **モデルを勝手に決めない**: 推奨は示すが、最終選択は必ずユーザーに確認する。
