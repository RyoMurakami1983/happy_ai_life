# Deep KPT reference

## 使う場面

- 問題が多かった
- 同じ問題が再発した
- 途中で戻りが発生した
- 原因が見えない
- 環境改善の種を強めたい
- skill 利用が複雑で、時系列をほどかないと整理できない

## 先にやること

KPT に入る前に、セッションストーリーを確認する。

1. 何をやろうとしたか
2. どの skill / hook / agent を使ったか
3. どこで出戻りや詰まりが起きたか
4. どの判断をやり直したか

## KPT

- **Keep**: 続けたい良い動き
- **Problem**: 直したい詰まりや摩擦
- **Try**: 次に試すこと

## アシスタント補助候補

ユーザーの Keep / Problem / Try が出たあと、アシスタント側でも補助候補を 1〜3 件だけ提示してよい。

- セッション中に観測したが、まだ言語化されていない良い動き
- 問題としては軽いが、次回も効きそうな摩擦
- Try に落とせる改善種

主役はユーザーのふりかえりなので、候補は上書きではなく補助として扱う。

### Try の扱い

Try は実験として書く。成功条件が見えるなら、T に落としてよい。

- Keep を伸ばす Try
- Problem を防ぐ Try
- 自動化・テンプレート化・導線改善の Try

## 深掘り

Problem が大きい場合は 5つのなぜで掘る。

1. 何が起きたか
2. なぜ起きたか
3. なぜそれが許されたか
4. どこで止められたか
5. どこを変えるべきか

その後、SMART に落とす。

## SMART

- Specific: 具体的に
- Measurable: 確認可能に
- Achievable: 実行可能に
- Relevant: 問題に直結させる
- Time-bound: 次回までに動ける形にする

## 改善 Issue 候補

KPT の Problem / Try から、追跡すべき改善だけを Issue 候補にする。

Issue 候補にする条件:

- 再発しそう、または複数 repo / 複数 session に効く
- 対象が skill / hook / docs / workflow / script のどれかに置ける
- Acceptance Criteria を観測可能に書ける
- 今すぐ直すより、追跡した方が安全

すべての Problem を Issue にしない。感想や単発の注意は session note に残す。

## skill 改善提案

skill 改善提案は常設項目ではない。次の条件を満たすときだけ出す。

- 問題の主因が skill / hook / docs / 導線の弱さにある
- その改善で同種の出戻りを減らせる
- SMART や Try だけでは吸収しきれない

書く場合も 1〜3 件に絞り、期待効果と対象を短く添える。

## 環境改善ループ

ふりかえりで見えた改善種は、次の順で回す。

1. Session note に残す
2. T / TRY に入れる
3. 必要なら `references/issue-discovery.md` の型で Issue 候補にする
4. skill / hook / docs の修正に繋げる

このループにすると、個人の学びが環境の強化に変わる。
