# ドメインモデリング

設計を進めながら、プロジェクトのドメインモデルを能動的に構築し、研ぎ澄ませてください。これは*能動的*な訓練です——用語に異議を唱え、エッジケースのシナリオを考案し、用語集と意思決定が固まった瞬間に書き留めます。（語彙のために`CONTEXT.md`を単に*読む*ことはこのスキルではありません——それはどのスキルでもできる習慣です。このスキルは、モデルを消費するだけでなく、変更するときのためのものです。）

## ファイル構成

ほとんどのリポジトリは単一のコンテキストを持ちます。

```

/

├── CONTEXT.md

├── docs/

│   └── adr/

│       ├── 0001-event-sourced-orders.md

│       └── 0002-postgres-for-write-model.md

└── src/

```

ルートに`CONTEXT-MAP.md`が存在する場合、そのリポジトリは複数のコンテキストを持ちます。マップは、それぞれの場所を示します。

```

/

├── CONTEXT-MAP.md

├── docs/

│   └── adr/                          ← システム全体の意思決定

├── src/

│   ├── ordering/

│   │   ├── CONTEXT.md

│   │   └── docs/adr/                 ← コンテキスト固有の意思決定

│   └── billing/

│       ├── CONTEXT.md

│       └── docs/adr/

```

ファイルは遅延的に作成してください——書くべき内容があるときだけ作成します。`CONTEXT.md`が存在しない場合は、最初の用語が確定した時点で作成してください。`docs/adr/`が存在しない場合は、最初のADRが必要になった時点で作成してください。

## セッション中に

### 用語集と照らし合わせて異議を唱える

ユーザーが`CONTEXT.md`内の既存の言語と矛盾する用語を使った場合、すぐに指摘してください。「あなたの用語集では『キャンセル』はXと定義されていますが、あなたが言っているのはYのようです——どちらですか？」

### 曖昧な言葉を研ぎ澄ませる

ユーザーが曖昧または多義的な用語を使った場合、正確な標準用語を提案してください。「『アカウント』と言っていますが、CustomerのことですかUserのことですか？それらは別のものです。」

### 具体的なシナリオを議論する

ドメインの関係性が議論されているとき、具体的なシナリオでストレステストしてください。エッジケースを探るシナリオを考案し、ユーザーに概念間の境界を正確にするよう迫ってください。

### コードと照合する

ユーザーが何かの動作について述べたとき、コードがそれと一致しているか確認してください。矛盾を見つけたら、それを提示してください。「あなたのコードはOrder全体をキャンセルしますが、あなたはたった今、部分キャンセルが可能だと言いました——どちらが正しいですか？」

### CONTEXT.mdをその場で更新する

用語が確定したら、その場で`CONTEXT.md`を更新してください。まとめて後で処理せず、発生した瞬間に記録してください。[CONTEXT-FORMAT.md](.\references\CONTEXT-FORMAT.md)の書式を使用してください。

`CONTEXT.md`は実装の詳細を一切含んではいけません。`CONTEXT.md`を仕様書、メモ帳、実装上の意思決定の保管場所として扱わないでください。それは用語集であり、それ以外の何物でもありません。

### ADRの提案は控えめに

以下の3つすべてが当てはまる場合にのみ、ADR（アーキテクチャ決定記録）の作成を提案してください。

1. **覆すのが難しい** — 後で考えを変えるコストが大きい
2. **文脈なしでは意外に思われる** — 将来の読み手が「なぜこうしたのか」と疑問に思うだろう
3. **本物のトレードオフの結果である** — 本当に複数の選択肢があり、特定の理由で一つを選んだ

3つのうちどれか一つでも欠ける場合は、ADRを作成しないでください。[ADR-FORMAT.md](.\references\ADR-FORMAT.md)の書式を使用してください。
