# ADR フォーマット

ADRは`docs/adr/`に置かれ、連番を使用します：`0001-slug.md`、`0002-slug.md`など。

`docs/adr/`ディレクトリは遅延的に作成してください——最初のADRが必要になったときだけ作成します。

## テンプレート

```md

# {意思決定の短いタイトル}

{1〜3文で：背景は何か、何を決めたか、なぜそうしたか。}

```

これだけです。ADRは一段落でも構いません。価値があるのは、意思決定が*なされたこと*と*その理由*を記録することであり、セクションを埋めることではありません。

## 任意のセクション

これらは本当に価値を加える場合にのみ含めてください。ほとんどのADRはこれらを必要としません。

- **Status**（フロントマター）（`proposed | accepted | deprecated | superseded by ADR-NNNN`）——意思決定が見直される場合に有用
- **Considered Options（検討した選択肢）** ——却下した代替案を記録しておく価値がある場合のみ
- **Consequences（影響）** ——非自明な下流への影響を指摘する必要がある場合のみ

## 採番

`docs/adr/`内を確認し、既存の最大番号に1を加えてください。

## ADRを提案すべきタイミング

以下の3つすべてが当てはまる必要があります。

1. **覆すのが難しい** — 後で考えを変えるコストが大きい
2. **文脈なしでは意外に思われる** — 将来の読み手がコードを見て「一体なぜこうしたのか」と疑問に思うだろう
3. **本物のトレードオフの結果である** — 本当に複数の選択肢があり、特定の理由で一つを選んだ

意思決定を覆すのが簡単なら、記録は不要です——どうせまた変えることになります。意外性がなければ、誰もなぜかと疑問に思いません。本当の代替案がなかったなら、「当たり前のことをした」以上に記録すべきことはありません。

### 該当するもの

- **アーキテクチャの形。** 「モノレポを使用している」「書き込みモデルはイベントソーシングで、読み取りモデルはPostgresに射影される」
- **コンテキスト間の統合パターン。** 「OrderingとBillingは同期HTTPではなく、ドメインイベントを介して通信する」
- **ロックインを伴う技術選定。** データベース、メッセージバス、認証プロバイダ、デプロイ先。すべてのライブラリではなく——置き換えに四半期かかるようなものだけ。
- **境界とスコープの意思決定。** 「顧客データはCustomerコンテキストが所有し、他のコンテキストはIDでのみ参照する」明示的な「しない」ことも、「する」ことと同じくらい価値があります。
- **明白な道からの意図的な逸脱。** 「Xという理由でORMではなく手動SQLを使用している」合理的な読み手が逆を想定するようなあらゆること。これらは、次のエンジニアが意図的だったものを「修正」してしまうのを防ぎます。
- **コードから見えない制約。** 「コンプライアンス要件のためAWSを使用できない」「パートナーAPI契約のため応答時間は200ms未満でなければならない」
- **却下が非自明な場合の却下された代替案。** GraphQLを検討して、繊細な理由でRESTを選んだ場合、それを記録してください——さもないと半年後に誰かがまたGraphQLを提案するでしょう。
