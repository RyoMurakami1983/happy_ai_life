# CONTEXT.mdフォーマット

## 構造

```md

# {コンテキスト名}

{このコンテキストが何であり、なぜ存在するのかを1〜2文で説明。}

## 用語

**Order（注文）**：

{その用語の1〜2文の説明}

_避けるべき語_：Purchase、transaction

**Invoice（請求書）**：

配送後に顧客へ送付される支払い請求。

_避けるべき語_：Bill、payment request

**Customer（顧客）**：

注文を行う個人または組織。

_避けるべき語_：Client、buyer、account

```

## ルール

- **見解を持つこと。** 同じ概念に複数の語が存在する場合、最良の一つを選び、他は`_避けるべき語_`として列挙してください。
- **定義は簡潔に保つこと。** 最大1〜2文まで。それが「何であるか」を定義し、「何をするか」は定義しないでください。
- **このプロジェクトのコンテキストに固有の用語のみを含めること。** 一般的なプログラミング概念（タイムアウト、エラー型、ユーティリティパターン）は、プロジェクトで多用されていても含めません。用語を追加する前に、「これはこのコンテキスト固有の概念か、それとも一般的なプログラミング概念か」を問うてください。前者のみが該当します。
- 自然なまとまりが生まれる場合は**用語をサブ見出しの下にグループ化**してください。すべての用語が単一のまとまった領域に属する場合は、フラットなリストで構いません。

## 単一コンテキストと複数コンテキストのリポジトリ

**単一コンテキスト（ほとんどのリポジトリ）：** リポジトリのルートに一つの`CONTEXT.md`。

**複数コンテキスト：** リポジトリのルートにある`CONTEXT-MAP.md`が、各コンテキストとその所在地、相互の関係を一覧化します。

```md

# コンテキストマップ

## コンテキスト

- [Ordering](./src/ordering/CONTEXT.md) — 顧客の注文を受け付け、追跡する
- [Billing](./src/billing/CONTEXT.md) — 請求書を発行し、支払いを処理する
- [Fulfillment](./src/fulfillment/CONTEXT.md) — 倉庫のピッキングと出荷を管理する

## 関係

- **Ordering → Fulfillment**：Orderingが`OrderPlaced`イベントを発行し、Fulfillmentがそれを消費してピッキングを開始する
- **Fulfillment → Billing**：Fulfillmentが`ShipmentDispatched`イベントを発行し、Billingがそれを消費して請求書を生成する
- **Ordering ↔ Billing**：`CustomerId`と`Money`の型を共有している

```

このスキルは、どちらの構造が該当するかを推測します。

- `CONTEXT-MAP.md`が存在する場合は、それを読んでコンテキストを特定する
- ルートに`CONTEXT.md`のみが存在する場合は、単一コンテキスト
- どちらも存在しない場合は、最初の用語が確定した時点でルートに`CONTEXT.md`を遅延的に作成する

複数のコンテキストが存在する場合、現在の話題がどのコンテキストに関連するかを推測してください。不明な場合は質問してください。
